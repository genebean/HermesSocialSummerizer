HermesSocialSummerizer MCP

A read-only MCP server that lets Hermes, Claude Code, or any MCP-capable agent fetch and summarize Mastodon, Bluesky, and Nostr feeds without exposing any social posting surface.

What it does

Reads social feeds

Fetch timelines and following feeds from Mastodon, Bluesky, and Nostr accounts configured in config.yaml.

Reads engagement history

Expose favourites, likes, boosts/reposts, bookmarks, reactions, and zaps so an agent can infer what is likely interesting.

Never posts

The MCP tool list contains no post, favourite, like, boost, follow, repost, or publish operation.

The server speaks MCP over stdio. In normal use, your MCP host launches it from .mcp.json or equivalent Hermes configuration.

Read-only guarantees

PlatformGuaranteeWhat that means
MastodonCredential-enforcedUse an OAuth token created with the read scope only. The Mastodon instance rejects writes from that token.
BlueskyCode-enforced or public-onlyWith an app password, this server only calls read API methods. Without an app password, it uses public AppView reads with no credential.
NostrStructuralOnly an npub public key is configured. Never put an nsec private key in this project.
Do not add write capabilities. This project is intentionally read-only. Do not add tools for posting, liking, favouriting, following, boosting, reposting, or publishing Nostr events.

Setup

For local development without Nix, install dependencies and create a local config file:

cd /path/to/HermesSocialSummerizer
npm install
cp config.example.yaml config.yaml

Create a .env file for secrets referenced by config.yaml:

MASTODON_MAIN_TOKEN=your_read_only_access_token
BSKY_PERSONAL_APP_PW=your_bluesky_app_password

The server loads .env automatically at startup. Keep both config.yaml and .env out of git.

Verify setup

npm run typecheck
npm test
npm run build

Run with Nix

The repo includes a flake that provides a dev shell, a production package, a container image, and a NixOS module.

Development shell

Use the dev shell when working in the repository. It provides Node 24, npm, and Nix formatting/linting tools.

cd /path/to/HermesSocialSummerizer
nix develop
npm install
npm run typecheck
npm test
npm run build

Production package

Build the production package with:

nix build

The resulting binary is:

./result/bin/social-reader

Run it by supplying a config path, cursor-state path, and required secret environment variables:

export SOCIAL_READER_CONFIG=/path/to/config.yaml
export CURSOR_STATE_PATH=/path/to/cursor_state.json
export MASTODON_MAIN_TOKEN=...
export BSKY_PERSONAL_APP_PW=...
./result/bin/social-reader

The binary still speaks MCP over stdio; point your MCP host at social-reader or /nix/store/.../bin/social-reader.

NixOS module

The flake exports:

nixosModules.default

The module writes a generated config containing only non-secret account metadata and ${ENV_VAR} placeholders. Actual secret values must come from the MCP host environment or systemd environment files.

{ inputs, ... }:
{
  imports = [ inputs.social-reader.nixosModules.default ];

  services.social-reader = {
    enable = true;
    user = "gene";
    dataDir = "/var/lib/social-reader";

    mastodon = [{
      id = "main";
      instanceUrl = "https://mastodon.social";
      accessTokenEnv = "MASTODON_MAIN_TOKEN";
    }];

    bluesky = [{
      id = "personal";
      handle = "example.bsky.social";
      appPasswordEnv = "BSKY_PERSONAL_APP_PW";
    }];

    nostr = [{
      id = "main";
      npub = "npub1...";
      relays = [ "wss://relay.damus.io" "wss://nos.lol" ];
    }];
  };
}

The module installs a social-reader wrapper on PATH. The MCP host can use command: social-reader, provided it also receives the secret env vars named in the generated config.

Run as a container

The flake can build an OCI/Docker-compatible image with the production social-reader binary.

nix build .#container
docker load < result

The image is named social-reader:latest. It still speaks MCP over stdio, so run it interactively with stdin/stdout connected to the MCP host.

Required runtime mounts and environment

Path/envPurpose
/config/config.yamlRead-only config file. Set SOCIAL_READER_CONFIG=/config/config.yaml.
/data/Writable cursor-state volume. Set CURSOR_STATE_PATH=/data/cursor_state.json.
Secret env varsMastodon tokens and optional Bluesky app passwords, usually from --env-file.

Example MCP host command

{
  "mcpServers": {
    "social-reader": {
      "command": "docker",
      "args": [
        "run", "--rm", "-i",
        "--env-file", "/path/to/.env",
        "-e", "SOCIAL_READER_CONFIG=/config/config.yaml",
        "-e", "CURSOR_STATE_PATH=/data/cursor_state.json",
        "-v", "/path/to/config.yaml:/config/config.yaml:ro",
        "-v", "/path/to/social-reader-data:/data",
        "social-reader:latest"
      ]
    }
  }
}
Container note: this is not an HTTP service. Do not publish ports. The MCP host starts the container as a subprocess and talks to it over stdio.

Configuration

config.yaml contains account definitions. Secrets should be referenced as environment variables instead of written directly.

mastodon:
  - id: main
    instance_url: "https://mastodon.social"
    access_token: "${MASTODON_MAIN_TOKEN}"

bluesky:
  - id: personal
    handle: "example.bsky.social"
    app_password: "${BSKY_PERSONAL_APP_PW}"
  - id: public-project
    handle: "example-project.bsky.social"
    app_password: ""

nostr:
  - id: main
    npub: "npub1..."
    relays:
      - "wss://relay.damus.io"
      - "wss://nos.lol"
SettingRequirement
mastodon[].instance_urlMust be an HTTPS URL.
mastodon[].access_tokenRequired. Use a read-scope token.
bluesky[].app_passwordOptional. Blank or omitted means public-only reads.
nostr[].npubMust start with npub1. Never use an nsec.
nostr[].relaysAt least one wss:// relay is required.

Override paths when needed:

export SOCIAL_READER_CONFIG=/path/to/config.yaml
export CURSOR_STATE_PATH=/path/to/cursor_state.json

Run the MCP server

The server runs over stdio, so it is normally launched by an MCP host. To sanity-check it manually:

node_modules/.bin/tsx src/server.ts

You will not see a friendly HTTP page; stdio is reserved for JSON-RPC. Use an MCP client or host to call tools.

Connect an MCP host

Claude Code / local MCP config

The repo includes .mcp.json. Ensure the paths match your checkout:

{
  "mcpServers": {
    "social-reader": {
      "command": "/path/to/HermesSocialSummerizer/node_modules/.bin/tsx",
      "args": ["/path/to/HermesSocialSummerizer/src/server.ts"]
    }
  }
}

Hermes Agent

mcp_servers:
  social-reader:
    command: /path/to/HermesSocialSummerizer/node_modules/.bin/tsx
    args: ["/path/to/HermesSocialSummerizer/src/server.ts"]

If the host does not load .env from the repo working directory, pass the required environment variables through the MCP host config.

Tool reference

Always call list_accounts first. Every account-specific tool takes an account_id returned by that call.

Account and lifecycle tools

ToolPurpose
list_accountsList configured account IDs and read-only guarantees.
reload_configReload config.yaml and rebuild clients without restarting the MCP host.
mark_seenAdvance one platform/account cursor after you have safely processed posts fetched with advance_cursor: false.

Common response-size controls

  • limit: defaults to 40 for Mastodon/Bluesky and 100 for Nostr; capped at 200.
  • max_pages: timeline catch-up pages; capped at 10.
  • max_content_length: truncates long text/content fields with an omitted-character suffix.
  • include_html: Mastodon only; raw HTML is omitted by default.

Cursor-safe usage

Timeline tools advance cursors by default. That is convenient for interactive reads, but automation should usually be more careful.

Safe automation pattern: call timeline tools with advance_cursor: false, persist or analyze the returned posts, then call mark_seen with the newest cursor value only after processing succeeds.
Platformmark_seen.cursor_value
MastodonNewest post ID string.
BlueskyNewest created_at ISO timestamp.
NostrNewest note created_at Unix timestamp.

Cursor state defaults to cursor_state.json in the repo. Set CURSOR_STATE_PATH to keep it elsewhere.

Daily digest usage

For social digest automation, pair this MCP with a Hermes-host collector such as HermesSocialDigestPipeline:

  1. Call list_accounts to discover accounts.
  2. Fetch timelines with advance_cursor: false.
  3. Persist compact candidates outside this repo.
  4. Call mark_seen after persistence.
  5. Use engagement-history tools to calibrate likely-interesting topics and authors.

This MCP remains the source of social data. The digest pipeline owns scheduling, local cache, ranking prompts, and generated reports.

Operations

Reload config without restarting

After editing config.yaml or environment variables, call:

reload_config

This rebuilds platform clients. It also closes old Nostr relay connections before replacing them.

Reset timeline cursors

Stop the MCP host, delete or move cursor_state.json, then restart. Or use CURSOR_STATE_PATH to keep per-environment state files.

Public-only Bluesky mode

Leave app_password blank or omit it. Timeline results are reconstructed from public follows rather than the authenticated algorithmic following feed.

Troubleshooting

SymptomWhat to check
MCP host cannot start the serverConfirm the command path points to node_modules/.bin/tsx and that npm install has been run.
Invalid config.yamlCheck HTTPS Mastodon URLs, wss:// Nostr relays, npub1 public keys, and missing required env vars.
No accounts returnedCheck SOCIAL_READER_CONFIG, config.yaml location, and top-level YAML keys.
Bluesky likes/reposts failThose tools require an authenticated Bluesky account. Public-only Bluesky mode cannot read private engagement history.
Repeated timeline itemsCheck whether automation is using advance_cursor: false without later calling mark_seen.
Unexpectedly empty timelineCursor state may already be advanced. Reset cursor_state.json or use a separate CURSOR_STATE_PATH for testing.
